# Audit Report — wow-class-tracker (2026-07-23)

## 1. Verdict

The tracker is in good health. The full suite passes (114/114), `npm run build` is clean on live data, the 07-22 nightly was verifiably textbook end-to-end (fetch → agents → gates → publish → digest → deploy), and the hardest-to-get-right machinery — Gate 0 boundaries, honest source typing under the WCL outage, the anomaly-ack firewall, era gating, the creator-layer scope firewall — all held up under adversarial verification. The known WCL rdps outage is being handled exactly as documented: honest `unreachable` rows, frozen `asOf` dates, a stable-fingerprint heartbeat alert, and no silent substitution. The defects found are real but contained: one **high** product-honesty gap (the 14-day-stale PTR empirical layer is invisible to site visitors), a cluster of **medium** issues split between a genuine monitoring blind spot (Archon numeric metrics silently stalled, ungated), a live projection-input bug (undated metaNotes win the "newest read" nudge), two UI readability problems, two documentation drifts that could mislead a future agent, and three product-completeness gaps. Nothing found threatens data integrity of what is published — the risks are of the "quiet degradation goes unnoticed" and "future session misled by stale docs" variety.

## 2. Confirmed issues

### High

**H1. The 07-09 WCL freeze is invisible on the published site** — `src/template.html` (product)
Every rdps-family cut (zone-54 testing, zone-56 M+ testing, Dummy Dome rDPS, live raid/M+ medians) is frozen at asOf 2026-07-09, yet the header reads "Latest snapshot: 2026-07-22" (template.html:1203 — `latestSnapshot()` over source page dates, murlok's 07-22 wins). The only stale cue is tiny per-metric `· 2026-07-09` mnotes (798, 835). PTR builds have landed since; a visitor reading the projection or Dummy Dome composite has no way to know the empirical layer is 14 days old. This is a site whose brand is honest data.
**Fix:** the build already knows the degraded state (degraded notes surface at build time). Emit a visible caveat strip when any required source exceeds its `maxAgeDays` — e.g. "Warcraft Logs rDPS data frozen at 2026-07-09 (upstream API outage)" — driven from the manifest/required-sources data, not hand-written. Pair with the suggested per-row stale styling (S-list) but the strip is the load-bearing part.

### Medium

**M1. Archon numeric metrics (160 rows) are ungated and silently stalled since 07-20** — `data/required-sources.json` (data-integrity)
"95th pct DPS/HPS (Mythic)", "Popularity", and "M+ score (95th pct)" all sit at asOf 2026-07-20 while every other Archon surface (tiers, encounter-tiers, survivability) refreshed to 07-22. Git trace shows nightly refreshes through 07-20, then nothing — and no gate noticed, because these families have no requirement rows.
**Fix:** two parts. (a) Add an `archon-metrics` requirement family to `data/required-sources.json` (source `archon`, name patterns for the four series, coverage-date semantics like the WCL families) so a stall reddens the heartbeat — this file is CODEOWNERS-protected, so it's a reviewed edit, which is correct. (b) Read the 07-21/07-22 refresh-metrics logs and agent transcripts to find why the fetch stopped (recipe drift vs upstream change) and fix the recipe; don't just add the gate and let it fail nightly.

**M2. Undated metaNote silently wins the "newest read" projection nudge** — `src/render.mjs` (~line 286) (code-quality)
`.sort((a, b) => String(b.date).localeCompare(String(a.date)))[0]` — a note with no date becomes `"undefined"`, which sorts above every ISO date descending. Verified by execution: an undated positive note beat a dated negative one and flipped the ±3 nudge. This is a live input to the published projection.
**Fix:** filter to dated notes before sorting (`.filter(n => n.date)`), and close the validator gap — require `date` on every `metaNotes[]` entry in `src/validate.mjs` (all current entries are dated, so this is safe to enforce). Add a unit test pinning that an undated note never wins.

**M3. 2026-07-23 nightly had not started as of 12:06 UTC; a fully dropped night wouldn't alert until 07-25** — (automation)
No run for 07-23 (newest is 07-22, green). Scheduler jitter has been 11:32–12:36 UTC the last four days, so this is most likely normal — but the freshness heartbeat's 36h threshold means a night that never fires isn't flagged until the following day's heartbeat.
**Fix:** verify tonight's run landed (it likely has by the time you read this). If dropped-night latency matters, tighten the heartbeat: since `startedAt` is full-precision, a ~26–28h threshold (cron interval + observed jitter + margin) would catch a dropped night the next morning instead of a day later. That's a one-number change in `data/required-sources.json`/`check-refresh.mjs` config — but weigh it against false-red risk from jitter; 30h is a safer first step.

**M4. Fight view renders the non-selected bracket as a column of bare em-dashes** — `src/template.html` (~712) (ui-ux)
With a raid boss selected, the M+ column shows "—" on all 40 rows — specs that read S a click earlier now read unrated — explained only by a hover tooltip, unreachable on touch.
**Fix:** when a fight is selected, either visually collapse/grey the other bracket's column with an in-column header note ("per-fight view — raid only"), or keep the consensus tier dimmed with a "(overall)" marker. A header-level note beats 40 identical tooltips.

**M5. Tier-set commentary leads with superseded numbers** — `src/template.html` (ui-ux)
Outlaw's box: datamined 4pc says "20% chance"; the commentary directly beneath opens "12% chance…" with the reconciling "(Pre-06-30 values — since raised to 20%…)" only at the end.
**Fix:** when `ptr.set2/set4` commentary predates `tierSet.asOf`, render the reconciliation note first (or as a prefix badge like "written pre-06-30 tuning"), not last. The dates needed for the comparison are already in the data; this is a template-only ordering change.

**M6. Robydoby's "intentionally not in required-sources.json" posture is documented nowhere** — `.claude/skills/refresh-metrics/SKILL.md` (docs)
The design decision (best-effort, deliberately outside the refresh contract) exists only in session memory. Two failure modes: a future session "fixes" the omission by proposing it into `required-sources.json`, or conversely lets the sheet rot with no one ever noticing because nothing at all watches it.
**Fix:** add one paragraph to SOURCES.md's robydoby row and the SKILL.md recipe stating the posture and why (community sheet, outage-prone, must never redden a night), plus a "check it opportunistically during metric refreshes; staleness is acceptable" instruction. Optionally a comment line in `required-sources.json` itself if the format allows.

**M7. watch-creators SKILL.md still frames the android-client transcript experiment as pending** — `.claude/skills/watch-creators/SKILL.md` (lines 93-99) (docs)
CLAUDE.md and SOURCES.md both record the workaround as failed (2026-07-17) with Supadata as the landed fallback; the skill an agent actually loads says "judge it on a few nights' logs… fallback decision is the owner's." A nightly agent following the skill verbatim could re-litigate a settled decision.
**Fix:** rewrite the gotcha to past tense: experiment failed 2026-07-17, Supadata captions API is the landed path (fetch-transcripts.mjs), runner-blocked videos queue for local catch-up. Five-minute edit.

**M8. History snapshots lack a season/settledness marker — the forecast report card's endpoint will be manual archaeology** — `src/snapshot.mjs` (product)
The enriched history is verifiably sufficient to grade projections, but issue #17's endpoint ("first settled S2 consensus") has no machine-readable marker; identifying it later means reading commit history by hand.
**Fix:** add an optional `meta` field to snapshots (e.g. `{ era: "12.0.7-live" | "12.1-live" }` derived from which partition the tier-list sources describe, or simply stamped when the owner flips a flag at S2 launch), and/or plan to create a one-time annotated tag at the freeze point. Cheap now, expensive to reconstruct later.

**M9. Digest is silent on encounter-tier and tier-set changes** — `src/digest.mjs` (lines 158-163) (product)
`payloadAt` hardcodes `encounterTiers: null`, so per-boss/per-dungeon Archon moves never reach the owner's daily email; `tierSet` rewording changes (the very thing the upkeep gate protects) are likewise unreported.
**Fix:** load `encounter-tiers.json` in `payloadAt` and diff per-fight tiers (same shape as the existing source-move diff); diff `spec.tierSet.asOf`/text and emit a "Tier sets updated: <specs>" line. Both are additive diff sections in an already-working diff engine.

**M10. PTR writeup backlog: 11/40 specs uncovered, including all of Warlock, and nothing tracks it** — `data/specs.json` (product)
Missing: Frost DK, Havoc DH, all three Druids' remaining specs (Feral/Guardian/Resto), Brewmaster, Holy Priest, Elemental, Affliction/Demonology/Destruction. These specs lose the ±7 verdict shift and fall to tuning-line classification — systematically lower-confidence forecasts, honest in the UI but invisible as a backlog.
**Fix:** add a coverage line to the digest health footer ("PTR writeups: 29/40; missing: …") so the gap is visible daily, and add a ptr-watch SKILL.md instruction to opportunistically source writeups for uncovered specs (Warlock first — a whole class at zero).
